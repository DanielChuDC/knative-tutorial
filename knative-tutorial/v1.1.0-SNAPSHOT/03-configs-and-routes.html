Configurations and Routes

At the end of this chapter you will be able to understand and know how to :

  • Deploy Knative configurations and routes separately.

  • Distribute traffic between revisions of a service.

Prerequisite

The following checks ensures that each chapter exercises are done with right environment settings.

  • Make sure to be connected to minikube or minishift docker daemon and using right openshift client.

eval $(minikube docker-env)
(OR)
eval $(minishift docker-env) && eval $(minishift oc-env)

  • Make sure right OpenShift and kubernetes client are used.

# kubernetes v1.12+
kubectl version
(OR)
# oc v3.11.0+0cbc58b kubernetes v1.11.0+d4cacc0
oc version

  • Make sure to be on knativetutorial OpenShift project/kubernetes namespace

# verify to return knativetutorial
oc project -q

If you are not on knativetutorial project, use the command oc project knativetutorial to change to knativetutorial project

Build Containers

If you have already cloned the repo and built the greeter service you can skip this step.

  • Navigate to work folder

cd $TUTORIAL_HOME/work

  • Clone the application source code

git clone https://github.com/redhat-developer-demos/knative-tutorial-greeter.git

  • Build the application and container image

cd knative-tutorial-greeter/java/springboot
./mvnw clean package
docker build -t dev.local/rhdevelopers/greeter:0.0.1 .

  • Check if images are built and available:

docker images | grep dev.local

The above command should return something like as shown below:

# dev.local/rhdevelopers/greeter   0.0.1               6aa8771da8dd        2 hours ago         456MB
You can also use the script buildImage.sh that will build source and create the container image

Deploy Configuration

In the previous chapter we saw how we can deploy services using a holistic service resource file, in this chapter we will see how to deploy the service using configurations and route files.

Navigate to the tutorial chapter’s knative folder:

cd $TUTORIAL_HOME/02-configs-and-routes

Deploy Configuration revision 1

The following snippet shows how a Knative configuration resource YAML looks like:

configuration-rev1.yaml
apiVersion: serving.knative.dev/v1alpha1
kind: Configuration
metadata:
  name: greeter
spec:
  revisionTemplate:
    metadata:
      labels:
        app: greeter
    spec:
      container:
        image: dev.local/rhdevelopers/greeter:0.0.1 (1)
1 It is very important that the image is a fully qualified docker image name, including a tag. For more details on this Question 2 of FAQ.

The service is then deployed using:

kubectl apply -n knativetutorial -f config/configuration-rev1.yaml
(OR)
oc apply -n knativetutorial -f config/configuration-rev1.yaml

After the deployment was successful, we should see a kubernetes deployment called greeter-00001-deployment.

Invoke Service

INGRESSGATEWAY=istio-ingressgateway
IP_ADDRESS="$(minikube ip):$(kubectl get svc $INGRESSGATEWAY --namespace istio-system --output 'jsonpath={.spec.ports[?(@.port==80)].nodePort}')"
curl -v -H "Host: greeter.knativetutorial.example.com" $IP_ADDRESS
(OR)
http $IP_ADDRESS 'Host: greeter.knativetutorial.example.com'
(OR)
INGRESSGATEWAY=istio-ingressgateway
IP_ADDRESS="$(minishift ip):$(oc get svc $INGRESSGATEWAY --namespace istio-system --output 'jsonpath={.spec.ports[?(@.port==80)].nodePort}')"
curl -v -H "Host: greeter.knativetutorial.example.com" $IP_ADDRESS
(OR)
http $IP_ADDRESS 'Host: greeter.knativetutorial.example.com'

The command will return HTTP 404 as there are no routes deployed yet. Let us now deploy a route.

Deploy Route

Let us now deploy a route that will route all the traffic to the configuration we just deployed.

route_default.yaml
apiVersion: serving.knative.dev/v1alpha1
kind: Route
metadata:
  name: greeter
spec:
  traffic:
    - configurationName: greeter
      percent: 100
kubectl apply -n knativetutorial -f route/route_default.yaml
(OR)
oc apply -n knativetutorial -f route/route_default.yaml

Invoking Service now should return a response like Hi greeter ⇒ greeter-00001-deployment-5d696cc6c8-m65s5: 1

Sometimes the response might not be returned immediately especially when the pod is coming up from a dormant state. You might need to be a little patient.

See what you have deployed

As we did not deploy a service this time

service

kubectl -n knativetutorial  get services.serving.knative.dev greeter
(OR)
# get a Knative Service (short name ksvc) called greeter
oc -n knativetutorial  get services.serving.knative.dev greeter

will throw an error as there are no knative services deployed as part of the configuration + routes based deployment strategy.

configuration

kubectl -n knativetutorial get configurations.serving.knative.dev greeter
(OR)
# get a Knative configuration called greeter
oc -n knativetutorial get configurations.serving.knative.dev greeter

routes

kubectl -n knativetutorial get routes.serving.knative.dev greeter
(OR)
# get a Knative routes called greeter
oc -n knativetutorial get routes.serving.knative.dev greeter

When the service is invoked with curl -H "Host: greeter.knativetutorial.example.com" $IP_ADDRESS, you noticed that we added a Host header to the curl command with the value greeter.knativetutorial.example.com. This FQDN is automatically assigned to your Knative service by the Knative route we created. It follows a format <route-name>.<namespace>.<domain-suffix>.

  • The domain-suffix (in this case example.com) is configurable via the config map config-domain in the knative-serving namespace.

revisions

kubectl -n knativetutorial get revisions.serving.knative.dev
(OR)
# get a Knative revisions called greeter you will see only one like greeter-00001
oc -n knativetutorial get revisions.serving.knative.dev

  • add -oyaml to the commands above to see more details

Deploy a new revision

Knative follows a revision model for its configurations. Each time you change something in the revisionTemplate Knative will effectively generate a new revision and thus a new deployment. In this example we add a new environment variable to demonstrate that mechanism.

Deploy configuration revision 2

configuration-rev2.yaml
apiVersion: serving.knative.dev/v1alpha1
kind: Configuration
metadata:
  name: greeter
spec:
  revisionTemplate:
    metadata:
      labels:
        app: greeter
    spec:
      container:
        image: dev.local/rhdevelopers/greeter:0.0.1
        env: (1)
          - name: MESSAGE_PREFIX
            value: Namaste
1 Adding an environment variable that will be used a message prefix.

Let us deploy the new revision using the command:

kubectl apply -n knativetutorial -f  config/configuration-rev2.yaml
(OR)
oc apply -n knativetutorial -f  config/configuration-rev2.yaml

After the deployment was successful we should see a kubernetes deployment called greeter-00002-deployment in the OpenShift dashboard:

Now running the command will show two revisions: greeter-00001 and greeter-0002.

Invoking Service will now show an output like Namaste greeter ⇒ greeter-00002-deployment-8d9984dc8-rgzx6: 2, where Namaste is the value that we configured via environment variable in the Knative service resource file.

Distributing traffic

When deploying services with service object based approach, the service resource will take care of traffic distribution either 100% to latest or pinned revision. In this section we will explore how we can deploy routes to distribute traffic between deployed revisions.

For the sake of clarity we will call greeter-00001 as revision 1 and greeter-00002 as revision 2

Before we start to apply routes, let us open a new terminal and run the following command $TUTORIAL_HOME/02-configs-and-routes/bin/call.sh, this command will keep sending requests to greeter route every two seconds which allows us to monitor the changes to the responses as we keep applying the route that will distribute the traffic between the available two revisions.

Send all traffic to revision 1

kubectl apply -n knativetutorial -f  route/route_all_rev1.yaml
(OR)
oc apply -n knativetutorial -f  route/route_all_rev1.yaml

You will notice the output on your monitoring terminal to be something like Hi greeter ⇒ greeter-00001-deployment-5d696cc6c8-m65s5: 34.

Send all traffic to revision 2

kubectl apply -n knativetutorial -f route/route_all_rev2.yaml
(OR)
oc apply -n knativetutorial -f route/route_all_rev2.yaml

You will notice the output on your monitoring terminal to be something like Namaste greeter ⇒ greeter-00002-deployment-8d9984dc8-rgzx6: 13.

50-50 split between revision 1 and revision 2

kubectl apply -n knativetutorial -f route/route_rev1-50_rev2-50.yaml
(OR)
oc apply -n knativetutorial -f route/route_rev1-50_rev2-50.yaml

You will notice the output will be mix of responses like Hi greeter ⇒ greeter-00001-deployment-5d696cc6c8-m65s5: 11 and Namaste greeter ⇒ greeter-00002-deployment-8d9984dc8-rgzx6: 10 approximately distributed 50% between the two.

75-25 split between revision 1 and revision 2

kubectl apply -n knativetutorial -f route/route_rev1-75_rev2-25.yaml
(OR)
oc apply -n knativetutorial -f route/route_rev1-75_rev2-25.yaml

You will notice the output will be mix of responses like Hi greeter ⇒ greeter-00001-deployment-5d696cc6c8-m65s5: 6 and Namaste greeter ⇒ greeter-00002-deployment-8d9984dc8-rgzx6: 7, with more requests responded by revision 1 approximately distributed 75% to 25 % between revision 1 and revision 2.

10-90 split between revision 1 and revision 2

kubectl apply -n knativetutorial -f route/route_rev1-10_rev2-90.yaml
(OR)
oc apply -n knativetutorial -f route/route_rev1-10_rev2-90.yaml

You will notice the will be mix of responses like Hi greeter ⇒ greeter-00001-deployment-5d696cc6c8-m65s5: 4 and Namaste greeter ⇒ greeter-00002-deployment-8d9984dc8-rgzx6: 5, with more requests responded by revision 2 approximately 10% to 90% between revision 1 and revision 2.

In the response texts e.g Hi greeter ⇒ greeter-00001-deployment-5d696cc6c8-m65s5: 4 the numbers at the end of the responses shows a count of how may requests have been handled by the service. In this example it is 4. Also note that the output count number may vary according to number of requests that you might have issued.

Cleanup

kubectl -n knativetutorial delete configurations.serving.knative.dev greeter
kubectl -n knativetutorial delete routes.serving.knative.dev greeter
(OR)
oc -n knativetutorial delete configurations.serving.knative.dev greeter
oc -n knativetutorial delete routes.serving.knative.dev greeter

Build

At the end of this chapter you will be able to:

  • Define build

  • Define build templates

  • How to make service use builds for building container images a.k.a source-to -url

Prerequisite

The following checks ensure that each chapter exercises are done with the right environment settings.

  • Make sure you connect to the minikube or Minishift Docker daemon and use the right kubectl or OpenShift client.

eval $(minikube docker-env)

(OR)

eval $(minishift docker-env) && eval $(minishift oc-env)

  • Kubernetes should be v1.12+

kubectl version

(OR)

  • OpenShift CLI should be v3.11.0+

oc version

  • Make sure to be on knativetutorial OpenShift project/Kubernetes namespace

oc project -q

If you are not on knativetutorial project, use the command oc project knativetutorial to change to knativetutorial project

Its required that you have container registry account which is needed during the exercises to push the built container images.

You can get one for free at Docker Hub

As the Knative build and build template specs need some customizations, such as creating a secret with your container registry credentials; we need to set the following environment variables before running the other commands of the section.

Navigate to the $TUTORIAL_HOME/05-build/knative folder:

cd $TUTORIAL_HOME/05-build/knative

Configure build environment variables

cp setenv.sh.example setenv.sh

Update the following environment variables as per your needs and customizations.

Table 1. Build Environment Variables
Variable Description Default Value e.g.

source $TUTORIAL_HOME/05-build/knative/setenv.sh

Edit Container Registry Secret

A container registry secret is required for the build to authenticate with the container registry, so that it can build and push the image to the registry.

yq w -i container-registry-secret.yaml 'metadata.annotations.[build.knative.dev/docker-0]' $REGISTRY_URL && \
yq w -i container-registry-secret.yaml stringData.username $REGISTRY_USERNAME && \
yq w -i container-registry-secret.yaml stringData.password $REGISTRY_PASSWORD

The above command will generate a Kubernetes secret as shown below:

container-registry-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  annotations:
    build.knative.dev/docker-0: https://index.docker.io/v1/
  name: basic-user-pass
stringData:
  password: demopassword
  username: demo
type: kubernetes.io/basic-auth
The above yaml has dummy values for username, password etc., the actual values will be based on the Build Environment Variables.

For more details on why we need to do this please check here.

Edit Knative build spec

The Knative build spec is used to create and run the Knative builds. You can find more details on Knative build here. In this example we will be using Buildah to perform the build and push of of the container image.

yq w -i event-greeter-build.yaml 'spec.steps[0].env[0].value' "/workspace/$CONTEXT_DIR" && \
yq w -i event-greeter-build.yaml 'spec.steps[0].workingDir' "/workspace/$CONTEXT_DIR" && \
yq w -i event-greeter-build.yaml 'spec.source.git.url' $SOURCE_REPO_URL && \
yq w -i event-greeter-build.yaml 'spec.source.git.revision' $SOURCE_REVISION && \
yq w -i event-greeter-build.yaml 'spec.steps[0].env[1].value' $DESTINATION_IMAGE_NAME

The above command will generate a knative build specification as shown below:

event-greeter-build.yaml
apiVersion: build.knative.dev/v1alpha1
kind: Build
metadata:
  name: greeter-build
spec:
  serviceAccountName: build-bot
  source:
    git:
      revision: master
      url: https://github.com/redhat-developer-demos/knative-tutorial-greeter.git
  steps:
    - args:
        - "/usr/local/bin/buildah.sh"
      env:
        - name: WORK_DIR
          value: /workspace/java/quarkus
        - name: DESTINATION_NAME
          value: quay.io/workspace7/build-test:v0.0.1
        # uncomment this if you have MAVEN_MIRROR thats is reachable from your kube cluster
        # speeds up the builds - the example below uses Nexus repo manager installed in host machine
        # - name: MAVEN_MIRROR_URL
        #  value: $http://192.168.64.1:8081/nexus/content/groups/public
      image: quay.io/rhdevelopers/quarkus-java-builder
      name: build-and-push
      # needed for buildah builds to be running as containers
      securityContext:
        capabilities:
          add: ["SYS_ADMIN"]
      volumeMounts:
        - mountPath: /root/.m2
          name: m2-cache
        - mountPath: /var/lib/containers
          name: container-storage
      workingDir: /workspace/java/quarkus
  timeout: 60m
  volumes:
    - name: m2-cache
      persistentVolumeClaim:
        claimName: m2-cache
    - name: container-storage
      hostPath:
        path: /var/lib/containers
The above yaml has dummy values for --context, --destination etc., the actual values will be based on the Build Environment Variables.

Edit Knative service

Run the following command to create the knative service that will use the image created by the event-greeter-build.

yq w -i service.yaml 'spec.runLatest.configuration.revisionTemplate.spec.container.image' $DESTINATION_IMAGE_NAME

service-build.yaml
apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: event-greeter
spec:
  runLatest:
    configuration:
      revisionTemplate:
        metadata:
          labels:
            app: event-greeter
        spec:
          container:
            image: docker.io/demo/event-greeter:0.0.1
The above yaml has dummy values for image, the actual values will be based on the Build Environment Variables.

Apply pre-requisite resources

Before we create the build and service, we need to create a few Kubernetes resources that will be used by the build.

Run the following commands to create the pre-requisite resources:

kubectl apply -n knativetutorial -f container-registry-secret.yaml && \(1)
kubectl apply -n knativetutorial -f build-sa.yaml && \(2)
kubectl apply -n knativetutorial -f m2-pvc.yaml  && \(3)
kubectl apply -n knativetutorial -f container-storage-pvc.yaml (4)

(OR)

oc apply -n knativetutorial -f container-registry-secret.yaml && \(1)
oc apply -n knativetutorial -f build-sa.yaml && \(2)
oc apply -n knativetutorial -f m2-pvc.yaml && \(3)
oc apply -n knativetutorial -f container-storage-pvc.yaml (4)

1 The Kubernetes secret called basic-auth-pass, check here for more details
2 The Kubernetes service account called build-bot that will run the build. Only build-bot has access to the secret basic-auth-pass. Check here for more details
3 The Kubernetes persistence volume claim used to cache maven artifacts, to make subsequent builds faster
4 The Kubernetes persistence volume claim that will be used to mount and store the built contianer images

Create build

kubectl apply -n knativetutorial -f event-greeter-build.yaml

(OR)

oc apply -n knativetutorial -f event-greeter-build.yaml

See what you have deployed

build

kubectl -n knativetutorial get builds.build.knative.dev event-greeter-build

(OR)

oc -n knativetutorial get builds.build.knative.dev event-greeter-build

The builds right now can’t be started from the step where it has failed, you always need to start a new build by deleting the existing build followed by creating a new one again.

e.g.

kubectl delete -n knativetutorial -f event-greeter-build.yaml
kubectl apply -n knativetutorial -f event-greeter-build.yaml

(OR)

oc delete -n knativetutorial -f event-greeter-build.yaml
oc apply -n knativetutorial -f event-greeter-build.yamll

Watching logs

The Knative build creates an init container for each build step. The init containers will be named like build-step-<step-name> e.g. if the build step is named "foo" then there will be an init container named "build-step-foo" added to the build pod.

The following command can be used to watch the log of a specific container within a pod,

kubectl logs -n {tutorial-namespace} <pod-name> -c <build-init-container-name>

(OR)

oc logs -n {tutorial-namespace} <pod-name> -c <build-init-container-name>

(e.g.)

To watch the container logs of container docker-push within the build pod called event-greeter-build-pod-2a271e:

kubectl logs -n {tutorial-namespace} event-greeter-build-pod-2a271e -c build-step-docker-push

(OR)

oc logs -n {tutorial-namespace} event-greeter-build-pod-2a271e -c build-step-docker-push

  • You can use the following command to list names of all the init containers within a pod .

(e.g.) the following command lists all the init container names of the pod event-greeter-build-pod-2986f2

kubectl -n {tutorial-namespace} get pods event-greeter-build-pod-2986f2 -o yaml | yq r - spec.initContainers[*].name

(OR)

oc -n {tutorial-namespace} get pods event-greeter-build-pod-2986f2 -o yaml | yq r - spec.initContainers[*].name

  • Using stern you can watch logs of all the container of the build pod using the command stern event-greeter-build-pod

Checking build status

A successful build will have the container image pushed to the container registry, going by the examples above you should the image `event-greeter:0.0.1 ` in your container registry.

You can check the status of build as shown below:

kubectl -n knativetutorial get pods

(OR)

oc -n knativetutorial get pods

The above command should return an output as shown below:

NAME                      READY     STATUS      RESTARTS   AGE
event-greeter-build-pod-cf0174   0/1       Completed   0          10m

With this you should also be able to do pull your container images from the registry.

Deploy service using Build

With a successful build you are ready to deploy the service, run the following commands to deploy the service:

kubectl apply -n knativetutorial -f service.yaml

(OR)

oc apply -n knativetutorial -f service.yaml

After successful deployment of the service we should see a Kubernetes deployment like event-greeter-8fvqf-deployment available.

Invoke service

INGRESSGATEWAY=istio-ingressgateway
IP_ADDRESS="$(minikube ip):$(kubectl get svc $INGRESSGATEWAY --namespace istio-system --output 'jsonpath={.spec.ports[?(@.port==80)].nodePort}')"

(OR)

INGRESSGATEWAY=istio-ingressgateway
IP_ADDRESS="$(minishift ip):$(oc get svc $INGRESSGATEWAY --namespace istio-system --output 'jsonpath={.spec.ports[?(@.port==80)].nodePort}')"

http POST $IP_ADDRESS \
  'Host:event-greeter.knativetutorial.example.com'  \
   message="test message"

The above command should return an response like:

{
    "host": "Event  greeter => 'event-greeter-build-pod-d55571' : 1",
    "message": "test message",
    "time": "04:50:40"
}
the actual response’s host and time properties will be based on your environment

Cleanup

Before proceeding next chapter let us delete the build and service objects that were created.

kubectl -n knativetutorial delete services.serving.knative.dev event-greeter
kubectl -n knativetutorial delete builds.build.knative.dev event-greeter-build

(OR)

oc -n knativetutorial delete services.serving.knative.dev event-greeter
oc -n knativetutorial delete builds.build.knative.dev event-greeter-build

You can also delete pre-req resources that were created if you don’t need them anymore.

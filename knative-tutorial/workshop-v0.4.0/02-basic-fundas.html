Knative Workshop

At the end of this chapter you will be able to :

  • Deploy a Knative service.

  • Deploy multiple revisions of a service.

  • Choose between different service deployment strategies.

  • Run different revisions of a service.

Prerequisite

The following checks ensure that each chapter exercises are done with the right environment settings.

  • OpenShift CLI should be v4.1+

oc version

  • Make sure to be on $WORKSHOP_USER-knativetutorial OpenShift project

oc project -q

If you are not on $WORKSHOP_USER-knativetutorial OpenShift project, then use the following command to change to $WORKSHOP_USER-knativetutorial project:

oc project $WORKSHOP_USER-knativetutorial

Deploy Service

Navigate to the tutorial chapter’s knative folder:

cd $TUTORIAL_HOME/02-basics/knative

The following snippet shows what a Knative service YAML looks like:

service.yaml
apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: greeter
spec:
  runLatest: (1)
    configuration:
      revisionTemplate:
        spec:
          container:
            image: quay.io/rhdevelopers/knative-tutorial-greeter:quarkus
1 Makes Knative always run the latest revision of the deployment

The service can be deployed using the following command:

oc apply -n $WORKSHOP_USER-knativetutorial -f service.yaml

After successful deployment of the service we should see a Kubernetes Deployment named similar to greeter-nsrbr-deployment available:

oc get deployments -n $WORKSHOP_USER-knativetutorial

  • The actual deployment name may vary in your setup

Invoke Service

SVC_URL="$(oc get ksvc greeter | awk 'NR==2{print $2}')"

curl -v $SVC_URL

The last http command should return a response containing a line similar to Hi greeter ⇒ '6fee83923a9f' : 1

Sometimes the response might not be returned immediately especially when the pod is coming up from dormant state. In that case, repeat service invocation.

See what you have deployed

The service-based deployment strategy that we did now will create many Knative resources, the following commands will help you to query and find what has been deployed.

service

oc -n $WORKSHOP_USER-knativetutorial  get services.serving.knative.dev greeter

configuration

oc -n $WORKSHOP_USER-knativetutorial get configurations.serving.knative.dev greeter

routes

oc -n $WORKSHOP_USER-knativetutorial get routes.serving.knative.dev greeter

  • The domain suffix in this case guru.devx.red is the OpenShift Router domain which gets automatically assigned once the Knative Service is created.

revisions

oc -n $WORKSHOP_USER-knativetutorial get rev -l serving.knative.dev/service=greeter --sort-by="{.metadata.creationTimestamp}"

  • add -oyaml to the commands above to see more details

Deploy a New Revision of a Service

As Knative follows 12-Factor application principles, any configuration change will trigger creation of a new revision of the deployment.

To deploy a new revision of the greeter service, we will add an environment variable to the existing service as shown below:

Service revision 2

service-env.yaml
apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: greeter
spec:
  runLatest:
    configuration:
      revisionTemplate:
        spec:
          container:
            image: quay.io/rhdevelopers/knative-tutorial-greeter:quarkus
            env:
             - name: MESSAGE_PREFIX (1)
               value: Namaste
1 Adding an environment variable that will be used as the message prefix

Let us deploy the new revision using the command:

oc apply -n $WORKSHOP_USER-knativetutorial -f service-env.yaml

After successful deployment of the service we should see a Kubernetes deployment called greeter-cqgcf-deployment.

Now if you list revisions

revisions

oc -n $WORKSHOP_USER-knativetutorial get rev -l serving.knative.dev/service=greeter --sort-by="{.metadata.creationTimestamp}"

You will see two of them, named similar to greeter-cqgcf and greeter-nsrbr.

Invoking Service will now show an output like Namaste greeter ⇒ '6fee83923a9f' : 1, where Namaste is the value we configured via environment variable in the Knative service resource file.

Pinning Service to a Revision

As you noticed, Knative service always routes traffic to the latest revision of the service deployment, due to the runLatest attribute in the service resource file.

Let us now make the greeter service use the first revision created, the one using the default MESSAGE_PREFIX.

You can use the show Knative service revisions command to find the available revisions for the greeter service.

Service pinned to first revision

service-pinned.yaml
apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: greeter
spec:
  release: (1)
    revisions:
    - greeter-nsrbr
  configuration:
    revisionTemplate:
      spec:
        container:
          image: quay.io/rhdevelopers/knative-tutorial-greeter:quarkus
          env:
            - name: MESSAGE_PREFIX
              value: Namaste
          livenessProbe:
            httpGet:
              path: /healthz
          readinessProbe:
            httpGet:
              path: /healthz
1 The release attribute in the service resource file will make Knative use the revision specified in the revisions attribute.

The names of revisions may be different in your setup

Let us redeploy the greeter service by pinning it to the first revision that was created:

PIN_REVISION=`oc get rev -n $WORKSHOP_USER-knativetutorial -l serving.knative.dev/service=greeter -l serving.knative.dev/configurationGeneration=1 | awk 'NR==2{print $1}'` && \
cat service-pinned.yaml | yq w - "spec.release.revisions[+]" $PIN_REVISION | oc -n $WORKSHOP_USER-knativetutorial  apply -f -

Invoking Service will produce output similar to Hi greeter ⇒ '6fee83923a9f' : 1

Cleanup

oc -n $WORKSHOP_USER-knativetutorial delete services.serving.knative.dev greeter
